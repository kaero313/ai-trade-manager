# AI-Trade-Manager 개발 운영 워크플로우

이 문서는 **Gemini 설계 + Codex 앱 적응형 멀티 에이전트 실행** 운영 모델을 정리합니다.
목표는 포트폴리오 품질을 해치지 않으면서도, 설계력·분해 능력·검증 습관·문서화·Git 활동성을 함께 드러내는 것입니다.

---

## 1. 운영 원칙
- **Gemini / IDE:** 기능 설계, 범위 확정, 수용 기준 작성, 코어 아키텍처/DB/외부 계약 변경 승인
- **Codex 앱:** 메인 실행 채널. Delta 판정, 작업 분해, 멀티 에이전트 실행, 검증, 문서 동기화, 커밋 담당
- **Codex CLI:** 좁은 단일 확인이나 반복 명령이 필요할 때만 사용하는 보조 채널

기본 흐름은 아래와 같습니다.
1. Gemini가 기존 마스터 프롬프트 형식으로 설계와 수용 기준을 작성한다.
2. 해당 프롬프트를 Codex 앱에 전달한다.
3. Codex 앱은 현재 리포지토리 기준으로 `이미 구현된 내용 / 남은 Delta / 충돌 여부`를 먼저 판정한다.
4. 독립 작업만 병렬화하고, 결과는 Main Orchestrator가 순차 통합한다.

---

## 2. Codex 앱 적응형 멀티 에이전트

### 2.1. 기본 역할
- **Main Orchestrator:** 항상 존재. 프롬프트 해석, Delta 판정, 작업 분해, 통합, 커밋, push 판단 담당
- **Explorer:** 현재 구현, 영향 범위, 충돌 가능성, 남은 Delta 조사 담당
- **Backend Worker:** API, 서비스, DB, 스케줄러, 비동기 로직 구현 담당
- **Frontend Worker:** React/Vite 화면, 컴포넌트, 상태 연동, 시각화 구현 담당
- **Reviewer:** 요구사항 누락, 구조 위반, 회귀 위험, 테스트 부족 검토 담당
- **Docs Curator:** README, `docs/ARCHITECTURE.md`, `docs/DATABASE.md`, `Agents.md` 동기화 담당

### 2.2. 작업 크기별 토폴로지
- **작은 작업:** `Main` 단독
  - 예: 단일 문구 수정, 국소 컴포넌트 보정, 작은 문서 수정
- **표준 작업:** `Main + Explorer + Worker 1명 + Reviewer`
  - 예: 단일 백엔드 기능, 단일 프론트 기능, 문서 포함 소규모 개선
- **포트폴리오 시그널이 큰 작업:** `Main + Explorer + Backend Worker + Frontend Worker + Reviewer`
  - 예: 새 대시보드 섹션, API+UI 연결, 스케줄러+시각화 연계
- **문서/설명 가치가 큰 작업:** 위 조합 + `Docs Curator`
  - 예: Phase 마감, 구조 변경, 포트폴리오 페이지 완성, AI 협업 흐름 정리

### 2.3. 기본 규칙
- 동일 파일을 동시에 수정하는 병렬 작업은 피한다.
- 하위 에이전트는 결과만 반환하고, 최종 통합과 커밋은 항상 Main이 수행한다.
- DB 스키마, 코어 아키텍처, 외부 API 계약 충돌이 감지되면 임의 우회 구현하지 않고 Gemini 재검토로 승격한다.
- `Reviewer 모든 작업 필수`는 적용하지 않으며, 중간 이상 작업에서 필수로 사용한다.

## 3. Gemini -> Codex 입력 계약
Gemini 마스터 프롬프트는 기존 형식을 유지하되, 아래 항목을 반드시 포함합니다.

IDE agent chat에서 Gemini 마스터 프롬프트를 생성할 때는 [IDE Agent Chat용 Codex 마스터 프롬프트 생성 템플릿](IDE_AGENT_CHAT_PROMPT_TEMPLATE.md)을 기본으로 사용합니다.
실무에서는 템플릿 전체를 그대로 넣고 마지막 `[사용자 요청]` 블록만 작업별로 교체한 뒤, IDE agent의 출력 본문 전체를 Codex 앱에 전달합니다.

- 목적
- 범위
- 수용 기준
- 제약 조건
- 문서 업데이트 요구
- 승인 필요한 변경 여부

권장 본문 구조는 아래와 같습니다.

```markdown
[Codex Master Prompt] Phase XX: 작업명

목적:
- 이 작업이 사용자와 시스템에 주는 가치

범위:
- 포함 범위
- 제외 범위

수용 기준:
- 완료로 볼 조건

제약 조건:
- 아키텍처, DB, API, 문서, 테스트 관련 제약

문서 업데이트:
- README / docs/ARCHITECTURE.md / docs/DATABASE.md / Agents.md 중 필요한 문서

승인 필요 여부:
- DB 스키마 변경 여부
- 코어 아키텍처 변경 여부
- 외부 계약 변경 여부

Task:
- 의미 단위 작업 목록
```

IDE agent가 작업 위험도를 높게 판단하거나 요구사항이 승인되지 않은 구조 변경을 포함한다고 보면, 정상 프롬프트 대신 `Gemini 재검토 및 프롬프트 갱신 요청` 형식으로 반환해야 합니다.

## 4. Codex 앱 실행 계약 래퍼
기존 Gemini 마스터 프롬프트 앞에는 아래 래퍼를 붙여 Codex 앱에 전달합니다.

```text
[Codex App Execution Wrapper]

아래는 Gemini가 작성한 마스터 프롬프트다.
먼저 현재 리포지토리와 비교해 이미 구현된 내용, 남은 Delta, 충돌/노후화 여부를 판정하라.
그 다음 의존 관계가 있는 작업만 순차 수행하고, 독립 작업만 병렬화하라.
동일 파일 동시 수정은 피하라.
DB 스키마, 코어 아키텍처, 외부 API 계약 변경은 영향 범위를 먼저 보고하라.
프롬프트에 결함이 있으면 임의 우회 구현하지 말고 Gemini 재검토 요청안을 작성하라.
관련 검증을 수행하고, README / docs/ARCHITECTURE.md / docs/DATABASE.md / Agents.md 중 필요한 문서를 동기화하라.
커밋은 의미 단위 마이크로 커밋으로 잘게 생성하되, 빈 커밋이나 설명 불가능한 커밋은 금지한다.
push 는 작업일 기준 하루 여러 번 수행할 수 있도록 작은 진척도 자주 정리하라.

[여기에 Gemini 마스터 프롬프트 본문]
```

## 5. Git 운영 규칙
Git은 프로젝트 포트폴리오 설명과는 별도로 **활동성 루틴**도 함께 고려합니다.

- 커밋은 `기능 + 문서 + 테스트 + 운영 규칙 정리`를 모두 허용한다.
- 커밋 메시지는 반드시 한국어 Conventional Commits를 사용한다.
- 커밋은 모델, 마이그레이션, 리포지토리, API, UI, 문서 단위로 잘게 나눈다.
- push 는 작업일 기준 하루 여러 번 수행하되, 의미 없는 빈 커밋은 금지한다.
- 잔디를 위해 커밋 수를 늘리더라도, 검증 불가능한 큰 덩어리를 한 번에 몰아 커밋하지 않는다.

권장 리듬은 아래와 같습니다.
1. 작업 시작 후 첫 의미 있는 진척에서 한 번 커밋
2. 중간 통합 시점에 한 번 더 커밋
3. 검증과 문서 동기화가 끝난 뒤 push

## 6. 검증 게이트
- **백엔드 변경:** `pytest`, `ruff`
- **프론트 변경:** `build`, `eslint`
- **크로스스택 변경:** 백엔드/프론트 검증 모두 수행
- **문서 변경:** 관련 코드/구조와 불일치 없는지 확인
- **중간 이상 작업:** Reviewer 검토 필수

검증 실패나 구조 충돌이 있으면 우회 구현하지 않고, 필요한 경우 Gemini 재검토 요청으로 승격합니다.
